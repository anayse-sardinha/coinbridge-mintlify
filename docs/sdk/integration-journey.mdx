Integrating CoinBridge involves two parallel tracks that must both be completed before going live: the **Mobile SDK** (embedded in your app) and the **Backend Connector** (your server-side integration). This guide walks you through every milestone, who owns it, and what CoinBridge needs to review before you proceed.

## Choose Your Track

<CardGroup cols={2}>
  <Card title="Mobile SDK" icon="mobile" href="/docs/sdk/implementation-roadmap">

## Full Integration Roadmap

The milestones below represent the complete integration path. Some can be worked in parallel. See the notes on each step.

<Steps>
  <Step title="Prerequisites & Environment Setup">
    **Owner:** Mobile Developer + Backend Developer

    **Estimated time:** [PLACEHOLDER: e.g., 1 week]

    Set up everything needed before writing a single line of SDK or connector code.

    **Mobile (SDK):**
    - Generate a 2048-bit RSA key pair and share the **public key** with CoinBridge
    - Install the SDK via Gradle (Android) or Swift Package Manager (iOS)
    - Configure NFC, Internet, and Biometric permissions in your app

    **Backend:**
    - Provision a server environment that meets the [SLA requirements](/docs/sdk/requirements/customer-technology) (99.96% uptime, 400ms response time for data-fetching endpoints)
    - Set up HTTPS endpoints to receive CoinBridge webhook calls

    <Note>
      Both tracks can begin in parallel from day one.
    </Note>
  </Step>

  <Step title="SDK Integration: Authentication and Initialization">
    **Owner:** Mobile Developer

    **Estimated time:** [PLACEHOLDER: e.g., 1–2 weeks]

    Implement the two foundational SDK calls that every session depends on.

    - Implement [`authenticate()`](/docs/sdk/installation/authentication): binds the user to the device using a JWT generated by your backend
    - Implement [`initialize()`](/docs/sdk/installation/initialize): syncs card state on every app launch
    - Handle session persistence and the re-login flow
    - (Optional) Implement [`isEligible()`](/docs/sdk/installation/user-eligibility) to gate access for ineligible users
  </Step>

  <Step title="SDK Integration: Card Issuance and Provisioning">
    **Owner:** Mobile Developer + QA Engineer

    **Estimated time:** [PLACEHOLDER: e.g., 1–2 weeks]

    Create the user's virtual card and provision it to their device wallet.

    - Implement [`issueCard()`](/docs/sdk/installation/issue-card): creates the permanent virtual card
    - Implement [`provisionCard()` / `addToApplePay()`](/docs/sdk/installation/add-to-wallet): pushes the card token to Google Pay or Apple Wallet
    - Test the full card setup flow end-to-end in the **sandbox environment**
    - Validate mandatory user communications (provisioning confirmation, abandoned flow follow-up)
  </Step>

  <Step title="Backend Connector: Data-Fetching Endpoints">
    **Owner:** Backend Developer

    **Estimated time:** [PLACEHOLDER: e.g., 1–2 weeks]

    Implement the endpoints CoinBridge calls in real-time during a transaction to authorize a payment.

    These are **time-critical**. They must respond within 400ms:
    - [`Get User Details`](/docs/webhooks/endpoints/get-user-details)
    - [`Get User Balance`](/docs/webhooks/endpoints/get-user-balance)
    - [`Get Spending Policies`](/docs/webhooks/endpoints/get-spending-policies)
    - [`Authorize Transaction`](/docs/webhooks/endpoints/authorize-transaction) *(if using the Customer Approves model)*

    <Note>
      This step can run in parallel with Step 3.
    </Note>
  </Step>

  <Step title="Backend Connector: Data-Reporting Endpoints">
    **Owner:** Backend Developer

    **Estimated time:** [PLACEHOLDER: e.g., 1 week]

    Implement the endpoints that receive transaction outcomes from CoinBridge after a payment is processed.

    - [`Report Transaction`](/docs/webhooks/endpoints/report-transaction)
    - [`Report Refund`](/docs/webhooks/endpoints/report-refund)
    - [`Report Correction`](/docs/webhooks/endpoints/report-correction)
    - [`Notify User Event`](/docs/webhooks/endpoints/notify-user-event)
  </Step>

  <Step title="Sandbox Testing">
    **Owner:** QA Engineer (+ Developer support)

    **Estimated time:** [PLACEHOLDER: e.g., 1–2 weeks]

    Validate the complete integration end-to-end before requesting CoinBridge review.

    - Test the full card setup flow (authenticate → initialize → issue → provision)
    - Simulate payment scenarios and verify transaction reports are received correctly
    - Test refund and correction flows
    - Validate error handling and edge cases (device ineligible, user not eligible, network errors)
    - Verify all mandatory communications are sent (Email/SMS for provisioning events)
  </Step>

  <Step title="CoinBridge Review: Sandbox Approval">
    **Owner:** Project Manager (coordinates submission)

    **Estimated time:** [PLACEHOLDER: e.g., review turnaround TBD by CoinBridge]

    <Warning>
      **Gating checkpoint.** You must receive explicit approval from CoinBridge before proceeding to production.
    </Warning>

    Submit the following materials to CoinBridge for review:
    - [PLACEHOLDER: list of required materials, e.g., test transaction logs, RSA public key confirmation, endpoint availability report]
    - [PLACEHOLDER: submission process / contact]

    CoinBridge will review and confirm before issuing production credentials.
  </Step>

  <Step title="Production Setup & Go-Live">
    **Owner:** Mobile Developer + Backend Developer + Project Manager

    **Estimated time:** [PLACEHOLDER: e.g., 1 week]

    Switch from the sandbox environment to production.

    - Replace sandbox credentials with production credentials in both the SDK and backend
    - Perform a final smoke test in production
    - Enable the CoinBridge feature flag for your user base
    - Monitor the first live transactions and Backend Connector response times

    <Note>
      Use a server-side feature flag to control rollout. This allows you to enable or disable CoinBridge remotely without a new app release.
    </Note>
  </Step>
</Steps>
